## 1. Дисковая подсистема TS-Conf

### 1.1. Что это такое

TS-Conf (TSConf) — ZX-Spectrum-совместимая платформа на FPGA. Дисковая подсистема —
это **SD-карта**, подключённая через так называемый **Z-Controller** (хардварный
SPI-контроллер). Поверх SD лежит файловая система **FAT32**. Под эмулятором Unreal
Speccy образ диска — это файл `wc.img` (сырой образ раздела/тома).

Ключевая мысль: на уровне железа вы работаете с **секторами по 512 байт** через
SPI-контроллер; на уровне ФС — с **кластерами, FAT-таблицей и каталогами**.

### 1.2. Z-Controller (SPI к SD-карте)

Доступ к SD идёт через два порта (используются в драйвере `sd_zc.a80`):

| Порт | Имя | Назначение |
|------|-----|------------|
| `#77` | config / chip-select | выбор карты (CS) + тактирование |
| `#57` | data | байт данных SPI (in/out) |

- **Chip Select:** запись в `#77` значения `%00000001` (`SD_CS1`) выбирает карту SD1;
  `%00000011` — снимает выбор (deselect).
- **Данные:** запись байта в `#57` отправляет его по SPI; чтение `#57` возвращает
  принятый байт. Контроллер тактирует SPI **аппаратно** — это критично (см. §1.4).
- Блочное чтение/запись делаются командой `INIR`/`OTIR` по порту `#57`: 256+256 байт
  на сектор (один `INIR` за `B=0` гоняет 256 байт).

### 1.3. SD-команды

Карта инициализируется самим Wild Commander (том уже смонтирован), поэтому плагину
достаточно команд чтения/записи:

| Команда | Код | Назначение |
|---------|-----|------------|
| CMD17 | `0x40+17` | чтение одного блока |
| CMD18 | `0x40+18` | чтение нескольких блоков (multi-block) |
| CMD24 | `0x40+24` | запись одного блока |
| CMD25 | `0x40+25` | запись нескольких блоков |
| CMD12 | `0x40+12` | стоп передачи (после multi-block) |

**Адресация.** Бывает двух видов, флаг лежит в глобале WC `SDBSF` (адрес `#6015`):
- **блочная** (SDHC/SDXC, `SDBSF != 0`): аргумент команды = номер сектора (LBA);
- **байтовая** (старые SDSC, `SDBSF == 0`): аргумент = `LBA * 512`.

Драйвер обязан учитывать флаг (в `sd_zc.a80` это `sd_blkt`).

### 1.4. Поллинг и частотная безопасность

Драйвер **нигде не использует цикловые задержки** — все ожидания построены на
поллинге статуса:
- ждать готовности: читать `#57`, пока не вернётся `#FF`;
- ждать data-token: читать `#57`, пока не `!= #FF` (затем должен прийти `#FE`/`#FC`);
- ждать окончания записи (busy): читать `#57`, пока не `!= 0`.

Поскольку Z-Controller тактирует SPI аппаратно, а ожидания — поллинговые, драйвер
**безопасен на любой частоте CPU** (3.5/7/14 МГц). Это тот же код, что и в самом WC
(`DSDZC.ASM`), а WC сам может работать на 14 МГц. **Вывод:** прямой доступ к SD
не требует подстройки под частоту, если построен на поллинге.

## 2. Файловая система FAT32

### 2.1. Геометрия тома (BPB)

Первый сектор тома — Boot/BPB. Поля, которые реально нужны (смещения в байтах):

| Смещ. | Размер | Поле | Заметка |
|-------|--------|------|---------|
| 11 | 2 | байт на сектор | всегда 512 |
| 13 | 1 | секторов на кластер (SPC) | степень двойки |
| 14 | 2 | зарезервированных секторов | до FAT |
| 16 | 1 | число копий FAT | обычно 2 |
| 21 | 1 | **media-дескриптор** | **`0xF0`** на SD/removable (см. §2.4) |
| 32 | 4 | всего секторов | 32-бит |
| 36 | 4 | размер FAT (FATSz32) | в секторах |
| 40 | 2 | ExtFlags | бит 7 = зеркалирование FAT выключено |
| 44 | 4 | кластер корня | обычно 2 |

Производные величины:
```
firstDataSector = reserved + numFATs * FATSz32
dataSectors     = totalSectors - firstDataSector
clusterSize     = SPC * 512
totalClusters   = dataSectors / SPC
```
LBA сектора кластера `c`: `firstDataSector + (c - 2) * SPC` (кластеры нумеруются с 2).

### 2.2. FAT-таблица и цепочки кластеров

FAT32 — массив 32-битных записей (используются младшие 28 бит). Запись `FAT[c]`
говорит, что идёт **после** кластера `c`:

| Значение `FAT[c]` (28 бит) | Смысл |
|----------------------------|-------|
| `0x0000000` | кластер **свободен** |
| `0x0000002 .. 0x?FFFFEF` | номер **следующего** кластера в цепочке |
| `0x?FFFFF7` | **bad cluster** (битый) |
| `>= 0x?FFFFF8` | **EOC** (End Of Chain — последний кластер файла) |

Файл — это цепочка кластеров: dir-запись хранит **первый** кластер, дальше идём по
FAT, пока не EOC. Свободное место = число записей `== 0`.

**Зарезервированные записи:**
- `FAT[0]` — media-дескриптор: `0x?FFFFF00 | media_byte`. На нашем томе `0xFFFFFFF0`.
- `FAT[1]` — EOC-маркер + флаги «чисто/грязно» (clean shutdown / hard error).
  Обычно `0xFFFFFFFF`.

Две копии FAT (FAT0 и FAT1) должны быть идентичны (зеркало). FAT0 — рабочая.

### 2.3. Каталоги

Каталог — цепочка кластеров с записями по 32 байта:

| Смещ. в записи | Поле |
|----------------|------|
| 0..10 | имя 8.3 (байт 0: `0x00` = конец каталога, `0xE5` = удалён) |
| 11 | атрибуты (`0x0F` = LFN, `0x08` = метка тома, `0x10` = каталог) |
| 20..21 | старшее слово первого кластера |
| 26..27 | младшее слово первого кластера |
| 28..31 | размер файла в байтах |

Длинные имена (LFN) — отдельные записи с атрибутом `0x0F` перед основной; символы
UTF-16 разбросаны по смещениям `1,3,5,7,9,14,16,18,20,22,24,28,30`.

### 2.4. ⚠️ Особенность: media-байт `0xF0`, а не `0xF8`

SD-карты и образы под Unreal — это **сменный носитель**, поэтому media-дескриптор
в BPB и в `FAT[0]` равен **`0xF0`** (removable), а НЕ `0xF8` (fixed disk), как на
обычных ПК-дисках. Значит:
- корректный `FAT[0]` тома = `0xFFFFFFF0`;
- любая проверка валидности `FAT[0]` обязана принимать `0xF0` (и `0xF8..0xFF`),
  иначе она ошибочно сочтёт здоровый том битым.

Этот нюанс однажды сломал зеркальный ремонт (см. §5.3) — запомните его.

## 3. Работа с устройствами через Wild Commander

### 3.1. Что такое Wild Commander

Wild Commander (WC) — двухпанельный файловый менеджер (Norton-подобный) для TS-Conf.
Он монтирует FAT32-том, рисует панели, обрабатывает клавиатуру, и предоставляет
плагинам **API** для работы с экраном, файлами и секторами. Точка входа API —
адрес `_WCAPI = #6006`: в `A` — номер функции, `CALL #6006`.

### 3.2. Полезные глобалы WC (область `#6000`)

| Адрес | Имя | Смысл |
|-------|-----|-------|
| `#6000..#6003` | `_PAGE0.._PAGE3` | номера страниц в окнах `#0000/#4000/#8000/#C000` |
| `#6004` | `_ABT` | флаг: нажат ESC |
| `#6005` | `_ENT` | флаг: нажат ENTER |
| `#6006` | `_WCAPI` | точка входа в API |
| `#6009` | `_TMN` | кадровый таймер (инкремент по прерыванию, ~50 Гц) |
| `#600D` | `CNFv` | версия видео/DAC-конфига (НЕ частота CPU) |
| `#6015` | `SDBSF` | флаг блочной адресации SD |

### 3.3. Основные функции API

(Номер в `A`, для функций с параметром — параметр в `A'` через `EXA`, см. §6.3.)

| Код | Имя | Назначение |
|-----|-----|------------|
| `0x00` | `MNGC_PL` | подключить страницу плагина в окно `#C000` (или экран/фонт) |
| `0x01` | `PRWOW` | вывести окно (структура SOW в `IX`) |
| `0x02` | `RRESB` | стереть окно (восстановить фон) |
| `0x03` | `PRSRW` | печать строки в окне (`HL`=текст в `#8000-#BFFF`!) |
| `0x04` | `PRIAT` | выставить цвет (после `PRSRW`) |
| `0x05` | `GADRW` | получить адрес в окне |
| `0x08` | `YN` | диалог OK/Cancel |
| `0x09` | `ISTR` | редактирование строки |
| `0x0A` | `NORK` | байт → две hex-цифры по адресу `HL` |
| `0x0B` | `TXTPR` | вывод многострочного текста в окне |
| `0x0D` | `DMAPL` | работа с DMA |
| `0x0E` | `TURBOPL` | смена частоты CPU/AY (см. §8) |
| `0x0F` | `GEDPL` | восстановить палитру/офсеты/текстовый режим |
| `0x10..0x17` | клавиши | SPACE, стрелки и т.п. |
| `0x30 / 0x31` | `LOAD512 / SAVE512` | чтение/запись 512-байтных блоков |
| `0x3B` | `FENTRY` | найти запись по имени |
| `0x3F` | `GDIR` | сделать найденный каталог активным |

> Текстовые управляющие коды в строках: `#0E` = центрировать, `#0D` = перевод строки.

### 3.4. Случайный доступ к секторам через WC

WC даёт доступ к произвольным секторам **текущего** устройства через свой
device-driver (исторически `LOAD512`/`SAVE512`, либо низкоуровневые thunk'и
`XPOZI`-seek + `RDDSE`/`SDDSE`). Это удобно: WC уже знает геометрию, номер
устройства и состояние SPI. Минус — зависимость от версии API.

В проекте ChkDsk мы пошли **напрямую** (свой `sd_zc.a80`), чтобы не зависеть от
тонкостей device-API и иметь полный контроль; WC при этом нужен только для отрисовки.

## 4. Прямой доступ к дисковым устройствам

Свой драйвер (`sd_zc.a80`) — это порт read/write-пути из `DSDZC.ASM` (WC). Логика:

**Чтение сектора (`sd_read_sector`):**
1. CMD17/CMD18 с адресом (LBA или LBA*512 по флагу).
2. Дождаться R1-ответа (бит 7 = 0).
3. Дождаться data-token `#FE`.
4. Прочитать 512 байт (`INIR`+`INIR`).
5. (для multi-block) CMD12 + досинхронизация + CS high.

**Запись сектора (`sd_write_sector`):**
1. CMD24/CMD25.
2. Послать data-token `#FC` + 512 байт (`OTIR`+`OTIR`) + 2 байта CRC (фиктивные).
3. Дождаться data-response, затем busy (карта держит `0`, пока пишет).
4. Стоп-token `#FD`, досинхронизация, CS high.

**Важно:** все ожидания — поллинговые, без задержек по тактам → драйвер
частото-независим (см. §1.4).

**Когда прямой доступ оправдан:** когда нужен полный контроль над секторами
(chkdsk, дамп, форматирование), независимость от device-API WC, или работа с FAT
напрямую (обход цепочек, переписывание записей FAT).

## 5. Ошибки FAT32 и их исправление

В проекте ChkDsk реализованы детект и ремонт пяти классов ошибок. Они делятся на
**детерминированные** (единственно верный исход — чиним автоматически) и
**best-effort** (исход неоднозначен — чиним «как лучше», возможна потеря данных).

### 5.1. Потерянные кластеры (lost clusters) — детерминированно

**Что это:** кластер помечен в FAT как занятый (`!= 0`), но ни один файл/каталог на
него не ссылается. Результат сбоя записи/краха.

**Детект:** `lost = allocated - used`, где `allocated = total - free`, а `used` —
сумма `ceil(размер/кластер)` по всем файлам + кластеры каталогов. Точный детект —
обход достижимости: построить bitmap кластеров, достижимых от корня; занятые, но
не отмеченные = lost.

**Ремонт:** обнулить `FAT[c]` (в обеих копиях) для каждого lost-кластера → они
становятся свободными. Данные не спасаются в файлы (в отличие от DOS `FOUND.NNN`),
просто возвращается место.

**Архитектура:** bitmap достижимости велик (на 201566 кластеров нужно ~25 КБ), но
адресное окно — 16 КБ. Поэтому ремонт **оконный**: bitmap покрывает
`WINDOW_CLUSTERS` кластеров (16 КБ = 131072 кластера), и дерево обходится заново для
каждого окна. На 201566 кластеров — 2 окна.

### 5.2. Рассинхрон копий FAT (backup-FAT mismatch) — детерминированно

**Что это:** FAT0 и FAT1 различаются (зеркало «протухло»).

**Детект:** посекторное сравнение FAT0 vs FAT1; число различающихся секторов.

**Ремонт (`repair_fat_mirror`, алгоритм dosfstools `read_fat`):**
- определить, какая копия «хорошая», по media-дескриптору `FAT[0]` (см. `media_ok`);
- `first_ok && !second_ok` → источник FAT0; `!first_ok && second_ok` → источник FAT1;
- иначе (обе валидны или обе битые) → по умолчанию FAT0 (первичная);
- скопировать различающиеся секторы источника поверх приёмника.

### 5.3. Битый заголовок FAT (`FAT[0]`/`FAT[1]`) — детерминированно (реконструкция)

**Что это:** повреждены служебные записи `FAT[0]` (media) или `FAT[1]` (EOC). Это
8 байт, не входящих ни в одну цепочку.

**Детект (`media_ok`, считается в скане для обеих копий):** `FAT[0]` low-byte ==
BPB media (см. §2.4!), байты 1-2 = `FF`, ниббл байта 3 = `F`; `FAT[1]` = EOC-маска.
Результат → `first_ok`/`second_ok`. Это **единственное место**, где видна ошибка
«обе копии битые одинаково» (тогда `mismatches = 0`, но заголовок невалиден).

**Ремонт:**
- одна копия валидна → восстановить вторую из неё (через `repair_fat_mirror`);
- **обе битые** → реконструировать из BPB: `FAT[0] = 0xFFFFFF00 | media`,
  `FAT[1] = 0xFFFFFFFF`, записать в обе копии. Источник истины — BPB, копия не нужна.

> 🐞 Исторический баг: `media_ok` был зашит на `0xF8` и не принимал `0xF0` →
> на любом SD-томе обе копии «невалидны» → ремонт всегда брал FAT0 как источник, и
> битая FAT0 затёрла бы хорошую FAT1. Исправлено принятием `F0` и `F8..FF`.

### 5.4. Несовпадение размера и цепочки (bad-size) — детерминированно

**Что это:** длина цепочки кластеров файла `!= ceil(размер/кластер)`.

**Детект:** при обходе цепочки файла сравнить её длину с ожидаемой из размера.

**Ремонт (безопасный, `walk_mode=3`):** выставить **размер = длина_цепочки ×
размер_кластера** (цепочка — авторитет для размещения). Трогаются **только записи
каталога**, FAT не меняется, кластеры не освобождаются → полностью безопасно.
Альтернатива (обрезать цепочку до размера) сложнее и трогает FAT.

### 5.5. Перекрёстные ссылки (cross-link) — best-effort (с потерей данных)

**Что это:** один кластер принадлежит двум+ цепочкам (два файла «делят» кластер и
всё, что идёт после него). Нельзя узнать, чьи это реально данные.

**Детект:** при обходе помечать кластеры в bitmap; если кластер уже помечен —
cross-link.

**Ремонт (`walk_mode=4`, best-effort):** первая цепочка, занявшая кластер (по порядку
обхода каталогов), его сохраняет; любая следующая, наткнувшаяся на него,
**обрезается** в предыдущем кластере (`FAT[prev] = EOC`, размер правится по
обрезанной длине). Если общий кластер — **первый** у второго файла, файл опустошается
(`start = 0, size = 0`). Усечённый файл теряет хвост — это и есть «best-effort».
Перекрёстных claim'ов и потерянных кластеров после ремонта не остаётся.

> Полностью восстановить данные при cross-link **нельзя в принципе** — нет источника
> истины. Можно лишь сделать FAT структурно консистентной.

### 5.6. Почему deep check (расширенная проверка) отдельно

Заголовок FAT (записи 0,1) и **данные** (записи 2+) — независимы. Битый заголовок —
почти косметика (media-байт advisory). А deep check обходит цепочки **данных** и
ищет cross-link/bad-size — реальную порчу файлов. Поэтому даже при `0 mismatches` и
валидном заголовке deep check полезен: `Cross-linked: 0  Bad-size: 0` подтверждает,
что сами файлы целы. Заголовок ≠ данные.

### 5.7. Порядок ремонта (как в dosfstools)

`FAT-mirror/заголовок → lost-кластеры → bad-size → cross-link → осиротевшие LFN`.
Сначала приводим FAT к консистентному виду, потом освобождаем мусор, потом правим
метаданные файлов, потом чистим осиротевшие записи каталога.

### 5.8. Осиротевшие записи длинного имени (orphaned LFN) — детерминированно

**Что это:** запись длинного имени (LFN, атрибут `0x0F`) — служебная, она имеет смысл
только как часть непрерывного «прогона» записей `0x0F`, идущего прямо перед короткой
8.3-записью-владельцем, чью контрольную сумму имени (стандартный FAT-чексум, см. ниже)
все записи прогона несут в смещении 13. Если такого живого владельца нет — записи
«осиротели».

**Источник:** баг удаления в Wild Commander. `CORE32.ASM`/`DELEN` при удалении
файла/каталога помечает удалённой (`0xE5`) **только** короткую запись, а
предшествующие LFN-записи оставляет нетронутыми. Поскольку WC генерирует LFN почти
для любого имени, не являющегося чистым 8.3 в верхнем регистре (в т.ч. **для всех
кириллических имён**), практически каждое удаление оставляет «прогон» осиротевших
`0x0F`. WC своей же ошибки не видит (перечислитель пропускает и `0xE5`, и `0x0F`),
зато её замечают Windows/chkdsk; вдобавок аллокатор WC дописывает новые записи после
первого `0x00` и не переиспользует `0xE5` — мусор копится.

**Чексум LFN** (тот же `LNCRC` в WC): `sum=0; для 11 байт имени: sum = ROR8(sum)+байт`
(в Z80 — `rrca`+`add`). Совпадение `sum` с байтом 13 прогона = прогон валиден.

**Детект (`walk_mode=5`, только чтение):** при обходе дерева вести «прогон» подряд
идущих `0x0F`. На терминаторе прогона решить:
- терминатор — живая короткая запись с совпавшим чексумом → прогон **валиден**
  (не трогаем — так валидные длинные имена остаются целы);
- терминатор `0xE5`/`0x00`/метка тома, либо чексум не совпал → прогон **осиротел**,
  прибавить его длину к счётчику `orphaned LFN entries`.
Быстро (только записи каталогов, без обхода цепочек) — считается сразу при скане, не в
deep check.

**Ремонт (`walk_mode=6`):** пометить `0xE5` первый байт каждой осиротевшей записи
`0x0F`. **FAT не трогается** (это только записи каталога) → полностью безопасно;
зануление мусорной записи необратимо лишь для самого мусора. Прогон может пересекать
границу 512-байтного сектора — поэтому межсекторный остаток несётся через `orph_pend`,
а хвост, оставшийся в предыдущем секторе, добивается перечитыванием `prev_lba`
(`fix_prev_cnt` записей с конца) — см. `fix_prev_lfn`.

## 6. Как написать плагин Wild Commander

### 6.1. Формат `.WMF` и заголовок

Плагин — файл `.WMF`, который WC грузит в память по дескрипторам из заголовка.
Сборка: `sjasmplus` → `SAVEBIN`. Структура заголовка (`wc_header.a80`):

| Смещ. | Поле |
|-------|------|
| +16 | сигнатура `"WildCommanderMDL"` |
| +32 | версия формата заголовка |
| +34 | **число страниц** (резервирует N×16 КБ, см. §7) |
| +35 | индекс страницы, подключаемой в `#8000` (обычно 0 = код) |
| +36 | дескрипторы загрузки: `db page, size512` (`size=0` — терминатор) |
| +160/+161 | макс. размер файла (0 = том-инструмент, не файловый) |
| +165 | имя плагина (32 байта) |
| +197 | триггер (`#03` = из меню плагинов) |
| +198.. | подпись пункта меню |

### 6.2. Загрузчик WC (как грузятся страницы)

Загрузчик (`WCVW.ASM`, рутина `WALK`):
- поле **+34** резервирует N 16-КБ страниц под плагин (**независимо** от
  дескрипторов загрузки);
- дескрипторы +36 обрабатываются последовательно: для каждого — подключить страницу
  через `MNGC_PL` в окно `#C000`, загрузить `size` 512-байтных блоков из файла туда;
  `size = 0` — конец списка;
- страница может быть **зарезервирована (через +34), но НЕ загружена** (поставить
  терминатор сразу после кода) → это чистый scratch на 16 КБ. Так ChkDsk кладёт
  свой 16-КБ bitmap на страницу `#C000`, не раздувая `.WMF`.

### 6.3. Точка входа и thunk'и API

Код плагина исполняется в окне `#8000-#BFFF`. Вызов API — `CALL #6006` с номером
функции в `A`. Для функций с параметром используется приём `EXA` (обмен `AF`/`AF'`):
параметр кладётся в `A`, потом `EXA` уводит его в `A'`, а в `A` ставится номер
функции. Стандартный thunk:
```
MNGC_PL: exa            ; параметр (номер страницы) -> A'
         xor a          ; A = 0 = код функции _MNGC_PL
         jp  _WCAPI      ; диспетчер вернёт параметр в A и вызовет функцию
```
Вызов: `ld a,<номер_страницы> : call MNGC_PL`.

### 6.4. Преимущества плагина перед прямым доступом к устройствам

- **Готовая инфраструктура:** WC уже смонтировал том, знает геометрию, нарисовал
  панели, обрабатывает клавиатуру. Не нужно писать загрузчик, монтирование, UI с нуля.
- **Готовый UI:** окна, строки, цвета, диалоги, редактор строки — через API.
- **Файловый доступ высокого уровня:** поиск записей, навигация по каталогам,
  `LOAD512`/`SAVE512` без ручной возни с SPI.
- **Интеграция:** запуск из меню, единый стиль, общий буфер с менеджером.
- **Безопасность частоты/кэша:** WC уже настроил CPU; драйвер на поллинге работает
  на любой частоте.

### 6.5. Недостатки и ограничения

- **Жёсткое ограничение памяти** — адресное окно всего 16 КБ под код+данные+буферы
  API (см. §7). Большие структуры (bitmap, кэши) приходится класть на отдельную
  страницу `#C000` и работать «окнами».
- **Зависимость от версии API** — номера функций и поведение могут отличаться между
  сборками WC. Недокументированные ветки (например restore-default в `TURBOPL`)
  использовать рискованно.
- **Прерывания WC** — таймер/часы продолжают тикать и трогать экранную страницу
  `#C000` (см. §7.3). Нужно аккуратно обращаться со страничным маппингом.
- **Нет прямого контроля над всем** — поведение монтирования, кэшей и т.п. задаёт WC.
- **Состояние CPU нельзя прочитать** — частота/кэш write-only (см. §8), нельзя «узнать
  и вернуть как было».

## 7. Память, страницы и прерывания

### 7.1. Окна памяти TS-Conf (16 КБ)

Адресное пространство Z80 (64 КБ) разбито на четыре окна по 16 КБ, каждое управляется
своим FPGA-регистром (порт `#xxAF`, старший байт = номер регистра):

| Окно | Адреса | Регистр (порт) |
|------|--------|----------------|
| Page0 | `#0000-#3FFF` | `#10AF` (`PW0`) |
| Page1 | `#4000-#7FFF` | `#11AF` |
| Page2 | `#8000-#BFFF` | `#12AF` (`PW2`) — здесь исполняется плагин |
| Page3 | `#C000-#FFFF` | `#13AF` (`PW4`) — сюда маппится scratch/экран |

Запись номера физической страницы в регистр окна подключает её в это окно.

### 7.2. Бюджет памяти плагина

- Код + переменные + буферы API живут в **`#8000-#BFFF`** (16 КБ). API-буферы
  (текст для `PRSRW`, структуры окон) **обязаны** быть в этом окне.
- Стек — в этом же окне (около `#BF00`).
- Большие scratch-структуры → на отдельную страницу, подключаемую в `#C000-#FFFF`
  (тоже 16 КБ). Плагину доступно до 63 страниц (`PLGPG..PLEPG`).
- `MNGC_PL`: `A` = индекс страницы плагина (0-based, <63) → подключить в `#C000`;
  `A = #FE` → текстовый экран WC; `A = #FF` → страница фонта.

Пример из ChkDsk: код ужат до ~7.5 КБ в `#8000`, а 16-КБ bitmap достижимости лежит
отдельной страницей на `#C000` (резерв через `+34=2`, маппинг через `MNGC_PL`,
очистка `bm_clear` в рантайме — из файла не грузится).

### 7.3. ⚠️ Прерывания и страница `#C000`

Прерывание WC (IM2) тикает таймер и **рисует часы**, обращаясь к экранной странице в
`#C000`. Если вы подключили туда свою страницу (bitmap), наивно — часы затрут ваши
данные. Спасает то, что обработчик часов (`HIME`) при каждом тике **сам**:
1. подключает системную страницу экрана в `#C000`,
2. рисует часы,
3. восстанавливает `#C000` по **теневому регистру `PGC` (`#6003`)**.

Поскольку `MNGC_PL` (через `MNGC`) обновляет `PGC`, ваша страница, подключённая
правильным API, **переживает тик прерывания** — часы вернут именно её. Поэтому:
- подключайте scratch-страницу через `MNGC_PL` (он пишет `PGC`) — и проход с битмапом
  interrupt-safe, `DI` ради этого не нужен;
- **только вокруг своих вызовов отрисовки** (`PRSRW`/`TXTPR`, которые пишут в экран
  по `#C000`) возвращайте экран (`MNGC_PL` с `#FE`), затем снова свою страницу.

В ChkDsk это сделано флагом `prog_swap`: во время оконного обхода (битмап в `#C000`)
прогресс-редрав свопает экран, а в фазах без битмапа (FAT-mirror, bad-size) — нет.

## 8. CPU: частота и кэш

- Частота CPU: регистр `SYC = #20AF`, младшие 2 бита → `0/1/2/3` = `3.5/7/14/28` МГц.
- Кэш: регистр `CYC = #2BAF` (значение по умолчанию `%00000010`).
- **Оба порта write-only.** WC хранит софт-тени (самомодифицируемые операнды) и
  никогда не читает их обратно → **прочитать текущую частоту/кэш при входе в плагин
  нельзя**. `CNFv` (`#600D`) — это версия видео/DAC, НЕ частота.
- Сменить частоту можно через API `TURBOPL` (`0x0E`): `B` = выбор (Z80/AY), `C` =
  значение; функция заодно переустанавливает кэш. Ветка restore-to-default
  (`B = #FF`) в текущей версии API **не документирована** → надёжно вернуть «как
  было» нечем.
- Для **тайминга** используйте `_TMN` (`#6009`) — кадровый счётчик прерывания
  (~50 Гц), **независимый от частоты CPU и кэша**. Для «эффективной скорости с учётом
  кэша» — самокалибровка: посчитать итерации цикла между двумя тиками `_TMN`.

**Практический вывод:** временно поднимать частоту на «расчёты» рискованно (нельзя
надёжно восстановить состояние, write-only порты), и для I/O-bound задач выгода мала.

## 9. Тестирование и отладка без железа

Подход проекта — **пирамида эмуляции**, чтобы не гонять каждую правку на железо:

- **Z80-симулятор (`sim_chkdsk.py`)** на основе чистого Python-ядра cburbridge:
  грузит `CHKDSK.WMF` по `#8000`, мокает `_WCAPI` как `RET`, эмулирует Z-Controller
  (порты `#77`/`#57`) поверх `wc.img`. Память плоская 64 КБ → `MNGC_PL` — no-op, но
  bitmap по `#C000` работает как обычная RAM. Позволяет вызывать рутины
  (`detect_walk`, `repair_*`, `bm_addr`-зонды) с лимитом инструкций → зависания
  всплывают как `TimeoutError`.
- **Маленький `test.img`** (~1000 кластеров, ~2 с) для быстрых прогонов; полный
  `wc.img` — для финальной проверки на реальной геометрии.
- **Скрипты порчи** для воспроизведения ошибок:
  - `make_lost_clusters.py` — потерянные цепочки в обоих окнах;
  - `make_fat_mismatch.py` — рассинхрон N секторов FAT1;
  - `make_fat0_corrupt.py` — невалидный media-дескриптор FAT0;
  - `make_fat_both_bad.py` — обе копии заголовка битые;
  - `make_deep_errors.py` — cross-link + bad-size (с сайдкаром для отката `--restore`).
- **Проверка «код не изменился»:** комментарии не влияют на кодоген → после правок
  комментариев `CHKDSK.WMF` должен остаться байт-в-байт прежним (сверка по хешу).

Чего симулятор НЕ проверяет: реальный страничный своп `#C000`, прерывания, тайминги
SPI на HW — это только на железе/в Unreal.

> Границы доверия эмуляторов: `MNGC_PL`/пейджинг, FM/DMA/IRQ, недокументированные
> опкоды, точный тайминг — не воспроизводятся. Render/DL-регрессии и
> phase-sensitive баги надёжнее ловить дампом с железа.

## 10. Справочник

### 10.1. Порты

| Порт | Назначение |
|------|------------|
| `#57` | SD data (SPI байт) |
| `#77` | SD config / chip-select |
| `#10AF/#11AF/#12AF/#13AF` | регистры окон Page0/1/2/3 (16 КБ) |
| `#20AF` | `SYC` — частота CPU (write-only) |
| `#2BAF` | `CYC` — кэш (write-only) |

### 10.2. Глобалы/адреса WC

| Адрес | Имя |
|-------|-----|
| `#6000-#6003` | `_PAGE0.._PAGE3` (номера страниц окон; `#6003` = `PGC`, тень `#C000`) |
| `#6004 / #6005` | `_ABT` (ESC) / `_ENT` (ENTER) |
| `#6006` | `_WCAPI` (вход в API) |
| `#6009` | `_TMN` (кадровый таймер) |
| `#600D` | `CNFv` (версия видео-конфига) |
| `#6015` | `SDBSF` (флаг блочной адресации SD) |

### 10.3. FAT32 значения записи

| Значение (28 бит) | Смысл |
|-------------------|-------|
| `0x0000000` | свободен |
| `0x0000002..0x?FFFFEF` | следующий кластер |
| `0x?FFFFF7` | bad cluster |
| `>=0x?FFFFF8` | EOC |
| `FAT[0]` | `0xFFFFFF00 \| media` (на SD: `0xFFFFFFF0`) |
| `FAT[1]` | `0xFFFFFFFF` (EOC + флаги) |

### 10.4. walk_mode в ChkDsk (обходы дерева)

| Режим | Назначение | Битмап |
|-------|------------|--------|
| 0 | подсчёт (файлы/каталоги/байты) | нет |
| 1 | пометить достижимые (для lost-ремонта) | да (`bm_set`) |
| 2 | детект cross-link + bad-size | да (`bm_mark_xlink`) |
| 3 | ремонт bad-size (размер ← цепочка) | нет |
| 4 | ремонт cross-link (обрезать 2-го) | да + запись FAT |
| 5 | детект осиротевших LFN (счёт `v_orphlfn`) | нет |
| 6 | ремонт осиротевших LFN (пометить `0xE5`) | нет (только записи каталога) |

### 10.5. Классы ошибок и тип ремонта

| Ошибка | Детект | Ремонт | Класс |
|--------|--------|--------|-------|
| lost clusters | allocated−used / достижимость | обнулить FAT | детерм. |
| FAT-mirror | посекторный diff | копия из хорошей | детерм. |
| битый заголовок | `media_ok` (обе копии) | из хорошей / из BPB | детерм. |
| bad-size | длина цепочки vs размер | размер ← цепочка | детерм. |
| cross-link | двойной claim в bitmap | обрезать 2-го | best-effort |
| осиротевшие LFN | прогон `0x0F` без живого владельца (чексум) | пометить `0xE5` | детерм. |
